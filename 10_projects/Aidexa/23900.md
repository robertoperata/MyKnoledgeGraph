# Task #23900 — Analisi: Closing Phase FAS + Registry + KYC

## Descrizione del task

```
GIVEN the onboarding has reached the final closing phase managed by the FAS service
WHEN the outbound request to the Registry service fails or is identified as unnecessary for the current contract
THEN the system must complete the KYC process and transition the onboarding to "Closed" status without displaying
```

---

## Glossario

- **FAS** = `financial-application-service`
- **KYC** = `know-your-customer`
- **BFF KYC** = `dynamic-onboarding-bff-kyc`
- **BFF Contract** = `dynamic-onboarding-bff-contract`
- **Centrico** = core banking system (sistema esterno)

---

## Cos'è la `/close`

La `/close` NON è un endpoint esposto dal FAS. È un endpoint di **Centrico**:

```
PUT /onboarding/v4.1/process/flow/{productName}/{guid}/close
```

### Chi la chiama

| Servizio                          | File                           | Metodo                                                            |
| --------------------------------- | ------------------------------ | ----------------------------------------------------------------- |
| `dynamic-onboarding-bff-contract` | `DigitalSigningService.kt:543` | `closeCentrico()` — chiamata dopo la firma digitale dei contratti |
| `bpm-conto-corrente`              | `CentricoWebClient.kt:64`      | `closeCentrico()` — service task Camunda                          |
| `bpm-x-instant`                   | `CentricoWebClient.kt:65`      | `closeCentrico()` — service task Camunda                          |

La chiamata nel BFF Contract avviene nella funzione `saveDigitalSigningData()`, condizionata dal feature flag `IS_CENTRICO_CLOSE_PROTECTION_ENABLED`.

---

## Come viene invocato il FAS nella fase di chiusura

Il contract BFF **non invoca direttamente il FAS** per la closing phase. Dopo `closeCentrico()`, il contract BFF fa:

```
closeCentrico()
  ↓
sendNewClientNetworkActivity()  → FAS /network-activities  (solo log IP per fraud detection)
  ↓
sendLastStepChanged()           → Azure Service Bus message (OnboardingStepChanged)
  ↓
thankyouPageReachedMessenger    → Azure Service Bus message (ThankyouPageReached)
  ↓
saveAndComplete()               → Camunda: getLastTask() + completeTask()
```

Il FAS è coinvolto solo per loggare l'attività di rete (`POST /network-activities`). Il completamento dell'onboarding transita attraverso **Camunda** (`saveAndComplete`), non attraverso il FAS direttamente.

La "closing phase managed by the FAS service" si riferisce a un **service task Camunda** (come avviene in `bpm-conto-corrente` e `bpm-x-instant`) che chiama il FAS, non a una chiamata diretta dal BFF contract.

---

## Il ruolo del FAS nella fase di chiusura

Il FAS viene invocato da un **delegate Camunda** (BPM service task) nella fase di chiusura. Il suo contributo è:

### 1. Chiama la Registry (Centrico)
- File: `DefaultApplicationProcess.kt:186` → `fetchAccountInfos(vatNumber)`
- Chiama: `centricoClient.searchRegistry(vatNumber)`
- Endpoint Centrico: `GET /core-banking/v4.0/registry/codes/{type}/{partita-iva}`
- Scopo: recuperare `subjectId` → poi chiamare `accountBySubject()` per trovare i conti bancari del cliente

### 2. Gestisce lo status dell'onboarding
- File: `DefaultApplicationProcess.kt:597`
- Legge `getFlowList()` da Centrico e mappa lo status via `externalFlowStatusToApplicationStatusCode()`
- File mapping: `adapter/shared/utils/mappers.kt:61`

### 3. Gestisce il KYC
- I callback Namirial passano per il FAS: `NamirialDigitalIdentificationCallbackController`
- Il servizio di callback è: `NamirialIdentificationCallbackService.handleStatusChanged()` (`kyc/NamirialIdentificationCallbackService.kt:29`)
- La persistenza KYC è in: `IdentificationRepositoryAdapter.upsertIdentification()`

---

## Gap attuali nel codice FAS

1. **`searchRegistry` non è fault-tolerant nel closing flow** — un suo fallimento non è gestito come condizione per completare il KYC (`DefaultApplicationProcess.kt:186-198`)
2. **`searchRegistry` viene chiamata sempre, indipendentemente dallo status** — ma il suo risultato è usato solo se l'application è in stato `DISBURSED` (vedi sezione Impatto sotto)
3. **Nessuna logica** che, al fallimento/inutilità della Registry, invochi la completion del KYC

## Flusso dettagliato KYC callback (punto 4)

```
Namirial
  │
  │  POST /applications/{onboardingId}/digital-identification/callback/session/{sessionType}/namirial
  ▼
NamirialDigitalIdentificationCallbackController.namirialCallback()
  │  converte il body in NamirialCallbackCommand via request.toCommand()
  │  passa onboardingId e sessionType al service
  ▼
NamirialIdentificationCallbackService.handleStatusChanged()
  │
  ├─① DB query → IdentificationRepositoryAdapter.findActivesByOnboardingIdAndStatusWithinDays()
  │    cerca la sessione con status="PENDING" creata negli ultimi 30 giorni per quell'onboardingId
  │    → se non trovata: errore DigitalIdentificationCallbackError
  │
  ├─② IdentificationStrategyFactory.getStrategy(callbackType)
  │    seleziona la strategy in base al sessionType (es. "SPID_CIE")
  │    → attualmente esiste solo DigitalIdentificationStrategy
  │
  ├─③ strategy.needToBeHandled(command.data.currentStatus)
  │    filtra i callback non rilevanti (solo "performed" viene gestito)
  │    → se false: ritorna Unit, nessuna scrittura su DB
  │
  ├─④ KycClientWebClientAdapter.fetchSessionInfo()
  │    GET /api/v1/session/{sessionId}?onboardingId=...&identificationType=...
  │    → chiama il servizio know-your-customer per recuperare i dettagli aggiornati della sessione
  │
  ├─⑤ strategy.resolveStatus()  [DigitalIdentificationStrategy]
  │    mappa currentStatus + dati sessione → IdentificationStatus:
  │
  │    "performed" + CIE      → CIE_COMPLETED
  │    "performed" + SPID     → COMPLETED (o MISSING_INFO se dati documento incompleti)
  │    "cancelled"            → CANCELLED
  │    altro                  → null (ignorato)
  │
  └─⑥ IdentificationRepositoryAdapter.upsertIdentification()
       pattern "chiudi e reinserisci":
       1. closeValidity() sul record padre esistente
       2. closeValidity() su document_data, residence_address, rejected_reasons
       3. insertActive() → nuovo snapshot con lo status risolto
       4. insertActive() → document data, residence address, rejected reasons (se presenti)
```

> **Nota**: questo flusso non notifica Camunda, non aggiorna lo status su Centrico, non triggera azioni esterne.
> Si limita a persistere lo stato KYC nel DB del FAS. Questo è il gap rilevante per il task: una volta
> completato il KYC, il sistema non fa nulla verso la closing phase.

## Flusso delle 4 chiamate dipendenti

Queste sono le 4 chiamate che compongono la catena osservabile nei log per un caso di chiusura:

```
POST /dynamic-onboarding-bff-contract/v1/digital-signing
    └─ applicationId
         └─► GET /know-your-customer/api/v1/session/{sessionId}
                  └─ onboardingId (confirmed) + status COMPLETED
                       └─► GET /financial-application/lending/{applicationId}
                                └─ businessKey (= partita IVA)
                                     └─► GET /external/centrico/core-banking/v4.0/registry/codes/partita-iva/{vatNumber}
                                              └─ subjectId, crCode, hostCodes
```

### Dettaglio di ogni chiamata

**1. `POST /dynamic-onboarding-bff-contract/v1/digital-signing`**
Avvia il processo di firma digitale. Produce una lista di contratti (`List<ContractDocument>`) con `id` e `barcode`. L'`applicationId` è il filo conduttore verso le chiamate successive.

**2. `GET /know-your-customer/api/v1/session/{sessionId}`**
Implementato in `KycDigitalIdentificationAdapter.kt`. Recupera i dettagli della sessione KYC (Namirial). Ritorna `status` (es. `COMPLETED`), `provider`, `externalSessionId`. Conferma che l'identificazione è completata prima di procedere.

**3. `GET /financial-application/lending/{applicationId}`**
Implementato in `LendingApplicationsController.kt:56`. Recupera i dettagli dell'application da BPM + Centrico. Ritorna `onboardingId`, `status`, `coreBankingDetails` e il campo chiave **`businessKey`** (= partita IVA del soggetto), che alimenta la chiamata successiva.

**4. `GET /external/centrico/core-banking/v4.0/registry/codes/partita-iva/{vatNumber}`**
Implementato in `CentricoClient.kt:122` tramite `searchRegistry()`. Riceve la partita IVA estratta dal passo precedente. Ritorna `subjectId`, `crCode`, `subjectHostCodes`. Il `subjectId` viene usato internamente per una seconda chiamata a `accountBySubject()`, che recupera i conti bancari del cliente.

---

## Impatto dell'eliminazione della chiamata #4

La chiamata #4 non è isolata: è usata **internamente** da `fetchAccountInfos()` (`DefaultApplicationProcess.kt:186`), che viene eseguita dentro la chiamata #3 (`GET /lending/{applicationId}`).

Il flusso interno è:
1. `searchRegistry(vatNumber)` → ottiene `subjectId`
2. `accountBySubject(subjectId)` → ottiene lista conti
3. Filtra conti con `accountType == "45"` (conto lending esterno)
4. `associateAccountToEntity()` abbina il conto all'application **solo se status == `DISBURSED`**

| Stato application | Senza chiamata #4 |
|---|---|
| `DISBURSED` | Dati di conto mancanti — `accountId` e `finalizedDate` sarebbero null nella response della #3 |
| Qualsiasi altro stato | Nessun impatto visibile — l'entità viene ritornata invariata |

**Conclusione:** la chiamata #4 è necessaria **solo per applicazioni in stato `DISBURSED`**. Per tutti gli altri stati (incluso il flusso di onboarding attivo dove avviene la firma digitale), la chiamata è inutile ma viene eseguita comunque.

## Modifica proposta: chiamare la Registry solo su stato DISBURSED

Nei due punti di `DefaultApplicationProcess.kt` dove viene invocata `fetchAccountInfos`, aggiungere un controllo sullo status prima della chiamata.

### Caso singolo — `retrieveFinancialApplicationDetails` (riga 376)

Lo status è già disponibile in `lendingApplicationDetails.status` prima della chiamata:

```kotlin
// PRIMA
val lendingAccounts = fetchAccountInfos(vatNumber).bind()
associateAccountToEntity(lendingApplicationDetails, lendingAccounts).bind()

// DOPO
if (lendingApplicationDetails.status == "DISBURSED") {
    val lendingAccounts = fetchAccountInfos(vatNumber).bind()
    associateAccountToEntity(lendingApplicationDetails, lendingAccounts).bind()
} else {
    lendingApplicationDetails
}
```

### Caso lista — `callCentricoAndPrepareResponse` (riga 145)

La chiamata avviene prima di sapere quali application nella lista sono `DISBURSED`. Basta un check preventivo:

```kotlin
// PRIMA
val lendingAccountIds = fetchAccountInfos(vatNumber).bind()
associateAccountsToEntities(applications, lendingAccountIds).bind()

// DOPO
val hasDisbursed = applications.any { it.status == "DISBURSED" }
val lendingAccountIds = if (hasDisbursed) fetchAccountInfos(vatNumber).bind() else emptyList()
associateAccountsToEntities(applications, lendingAccountIds).bind()
```

### Considerazioni

- **Nessun rischio logico**: `associateAccountToEntity` già ignora gli account se lo status non è `DISBURSED` — il comportamento finale è identico
- **Retrocompatibile**: nessuna firma API cambia, nessun test di integrazione dovrebbe rompersi
- **Separazione netta**: questa modifica risolve il problema "Registry inutile per contratti non DISBURSED", indipendentemente dal problema di fault-tolerance
