# Task #22927 — [VI] Send creditApplicationFeePercentage null to Actico when it should be calculated by Actico

## Descrizione del task

```
GIVEN a funding proposal with the feature flag isDifferentiatedInquiryFeeEnabled = true
WHEN no discount has been applied to the inquiry fee (discountedInquiryRate == originalInquiryRate)
THEN creditApplicationFeePercentage sent to Actico must be null,
     so that Actico calculates the fee autonomously instead of receiving it from us
```

---

## Glossario

- **Actico** = motore di pricing esterno (credit decision engine)
- **BFF Product** = `dynamic-onboarding-bff-product`
- **Camunda / BPM** = orchestratore di processo
- **inquiryFeePercentage** = percentuale commissione istruttoria
- **creditApplicationFeePercentage** = campo inviato ad Actico nella request di pricing
- **isDifferentiatedInquiryFeeEnabled** = feature flag in `START_CONFIG_FLAGS` su Camunda
- **InquiryRates** = oggetto con `originalInquiryRate` e `discountedInquiryRate`

## Logica della modifica (in `pricingAndSustainabilityResponse`)

```kotlin
val startConfigFlags = bpmWebClientAdapter.getStartConfigFlags(bpmId).getOrNull()

val inquiryFeePercentage =
    if (
        startConfigFlags?.isDifferentiatedInquiryFeeEnabled == true &&
            inquiryRates.discountedInquiryRate == inquiryRates.originalInquiryRate
    ) {
        null   // Actico calcola lui la commissione
    } else {
        inquiryRates.discountedInquiryRate  // mandiamo noi il valore
    }

acticoAdapter.buildPricingSustainabilityRequest(
    ...
    inquiryFeePercentage = inquiryFeePercentage,
    ...
)
```

**Condizione doppia (importante):**
- `isDifferentiatedInquiryFeeEnabled == true` → flag abilitato per questo onboarding
- `discountedInquiryRate == originalInquiryRate` → nessuno sconto è stato applicato

Se è stato applicato un codice sconto, i due tassi differiscono e il `null` NON viene mandato anche se il flag è attivo.

## Come funziona `getInquiryRates`

```
originalInquiryRate  = tasso base del prodotto (es. 0.015)

inquiryFeeDiscountCode = null   → sconto = 0   → discounted == original  ✓ null ad Actico
inquiryFeeDiscountCode = "XYZ"  → sconto > 0   → discounted < original   ✗ valore ad Actico
```

---

## Dove vive il flag

Il flag `isDifferentiatedInquiryFeeEnabled` è letto da Camunda tramite:

```kotlin
// BpmWebClientAdapter.kt:83
suspend fun getStartConfigFlags(bpmId: String) =
    loadOptionalJsonVariable<StartFeatureFlag>(bpmId, TaskVariable.START_CONFIG_FLAGS.value)
```

Variabile Camunda: `START_CONFIG_FLAGS` (JSON).

---

## Gotcha

La responsabilità della decisione `null` vs `discountedRate` è in **`FundingProposalServiceV2`**, non in `ActicoAdapter`. L'adapter riceve già il valore finale e lo passa direttamente come `creditApplicationFeePercentage`. Un test che mocka `isDifferentiatedInquiryFeeEnabled` sull'adapter è errato: il flag non viene letto lì.

## Test aggiunto — `FundingProposalServiceV2Test`

### Perché era necessario

La pipeline di Azure falliva per **code coverage insufficiente** (branch coverage). Il nuovo codice in `pricingAndSustainabilityResponse` introduce un `if/else` con due branch:

| Branch | Condizione | Coperta da test preesistenti? |
|---|---|---|
| `discountedRate` inviato | flag `false` oppure sconto applicato | ✓ sì (`@BeforeEach` mocka flag = false) |
| `null` inviato | flag `true` AND nessuno sconto | ✗ no → **aggiunto** |

### Nome del test

```
when isDifferentiatedInquiryFeeEnabled is true and no discount applies,
creditApplicationFeePercentage sent to Actico is null
```

### Setup dei mock — come si arriva al branch `null`

Il test usa i mock del `@BeforeEach` come base e sovrascrive solo quelli necessari:

**1. Flag abilitato**
```kotlin
whenever(bpmWebClientAdapter.getStartConfigFlags(any()))
    .thenReturn(StartFeatureFlag(isDifferentiatedInquiryFeeEnabled = true).right())
```
→ Sovrascrive il default del `@BeforeEach` che ritornava `StartFeatureFlag()` (flag = false).

**2. Sconto azzerato**
```kotlin
whenever(inquiryFeeDiscountsService.findByCode(any(), any(), any()))
    .thenReturn(BigDecimal.ZERO.right())
```
→ Il `@BeforeEach` mockava un discount di `0.0025`. Azzerandolo:
```
discountedInquiryRate = 0.015 - 0.0 = 0.015 == originalInquiryRate (0.015)
```
→ La condizione `discounted == original` diventa vera.

> **Nota:** il codice sconto `"ISTRUTTORIA2023"` è **presente** (viene dal JSON di test caricato nel `@BeforeEach`), ma il suo valore economico è 0. Questo simula il caso reale: cliente con flag differenziato attivo ma senza sconto applicabile.

**3. Flag lending discounting disabilitato**
```kotlin
whenever(appConfigService.getIsFeatureFlagEnabled(AppConfigFeatureToggles.LENDING_DISCOUNTING_ENABLED))
    .thenReturn(false)
```
→ Evita di entrare nel branch `creditDecisionAdapter` (non oggetto di questo test), tenendo il flusso sul percorso standard BPM.

### Come viene verificato il risultato

Dentro il `doAnswer` del mock di `buildPricingSustainabilityRequest`, il test cattura l'argomento `inquiryFeePercentage` (posizione 3) prima che venga usato:

```kotlin
var capturedInquiryFeePercentage: BigDecimal? = BigDecimal.ONE // sentinel value

whenever(acticoAdapter.buildPricingSustainabilityRequest(...))
    .doAnswer {
        capturedInquiryFeePercentage = it.arguments[3] as BigDecimal?
        acticoPricingSustainabilityRequest.copy(...).right()
    }

fundingProposalService.getFundingProposal(BPM_ID).assertIsRight()

assertNull(capturedInquiryFeePercentage,
    "creditApplicationFeePercentage should be null when " +
    "isDifferentiatedInquiryFeeEnabled=true and no discount is applied")
```

Il valore iniziale `BigDecimal.ONE` è un **sentinel**: se il `doAnswer` non venisse mai eseguito (es. mock non raggiunto), l'assert fallirebbe comunque — distingue "non chiamato" da "chiamato con null".

```

  │    ├─ loadTaskVariableStepFundingRequest → "ISTRUTTORIA2023" (da JSON)
```

## Test fixato — `ActicoAdapterTest`

### Problema

Il test `when differentiated inquiry fee flag is enabled, pricing request creditApplicationFeePercentage is not sent` passava `0.015` come `inquiryFeePercentage` ma si aspettava `null` come `creditApplicationFeePercentage`, mockando `isDifferentiatedInquiryFeeEnabled = true`.

Falliva con:
```
java.lang.AssertionError: actual value is not null expected null, but was:<0.015>
```

### Causa

`ActicoAdapter.buildPricingSustainabilityRequest` non legge `isDifferentiatedInquiryFeeEnabled`. Passa l'`inquiryFeePercentage` ricevuto direttamente come `creditApplicationFeePercentage`. La decisione di inviare `null` avviene **upstream** in `FundingProposalServiceV2`.

### Fix

```kotlin
// PRIMA — errato: passa 0.015 e si aspetta null basandosi sul flag
acticoAdapter.buildPricingSustainabilityRequest(
    ...
    0.015.toBigDecimal(),   // ← non diventerà mai null nell'adapter
    ...
)

// DOPO — corretto: il null arriva già dall'upstream
acticoAdapter.buildPricingSustainabilityRequest(
    ...
    null,   // ← FundingProposalServiceV2 ha già deciso
    ...
)
```

Il mock di `getStartConfigFlags` è stato rimosso dal test perché irrilevante: l'adapter non lo chiama.

## Dove si inseriscono i test nella sequenza

### Spiegazione testuale (pseudo-code)

La catena reale a runtime esegue questo:

```
1. FundingProposalServiceV2.pricingAndSustainabilityResponse()
   │
   ├── 2. chiede a Camunda: getStartConfigFlags(bpmId)
   │         └── risponde: { isDifferentiatedInquiryFeeEnabled: true }
   │
   ├── 3. ha già in mano inquiryRates da getInquiryRates():
   │         └── original = 0.015, discounted = 0.015  (nessuno sconto)
   │
   ├── 4. valuta la condizione:
   │         isDifferentiatedInquiryFeeEnabled == true  → ✓
   │         discounted == original                      → ✓
   │         quindi: inquiryFeePercentage = null          ← DECISIONE
   │
   └── 5. chiama ActicoAdapter.buildPricingSustainabilityRequest(
               inquiryFeePercentage = null   ← null già deciso al passo 4
           )
               └── 6. costruisce ActicoPricingSustainabilityRequest(
                           creditApplicationFeePercentage = null  ← passthrough
                       )
                           └── 7. invia HTTP ad Actico (campo assente)
```

**`FundingProposalServiceV2Test` (aggiunto)** — testa dal punto 1 al punto 5 incluso.
Verifica che la decisione al punto 4 produca `null` che arriva al punto 5. L'adapter (punti 6-7) è mockato.

```
Test entra qui → [1. pricingAndSustainabilityResponse]
                      ↓
                 [2. getStartConfigFlags]  ← mockato → true
                      ↓
                 [3. inquiryRates]         ← mockato → 0.015 == 0.015
                      ↓
                 [4. DECISIONE → null]
                      ↓
                 [5. buildPricingSustainabilityRequest(null)]  ← qui cattura e verifica
                      ↓
                 [6-7. ADAPTER + HTTP]     ← mockato, non eseguito
```

**`ActicoAdapterTest` (fixato)** — testa solo il punto 6.
Assume che il punto 4 sia già avvenuto upstream e che `null` arrivi già pronto. Verifica solo che venga messo nel campo giusto della request HTTP.

```
[1-5. FundingProposalServiceV2]  ← NON esiste, siamo fuori dal service

Test entra qui → [6. buildPricingSustainabilityRequest(null)]
                      ↓
                 verifica: creditApplicationFeePercentage == null ✓
                      ↓
                 [7. HTTP ad Actico]  ← mockato
```

**Il bug del test originale**: passava `0.015` invece di `null` al punto 6, assumendo che il punto 6 rileggesse il flag da Camunda e decidesse lui. Ma il punto 6 non sa nulla del flag — riceve e basta.
